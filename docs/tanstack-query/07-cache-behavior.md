# キャッシュ挙動の完全解説

TanStack Query のキャッシュは「自動的にデータが最新になる」ものではありません。
このドキュメントでは、**データがいつ取得され、いつ更新され、いつ破棄されるか** を正確に解説します。

---

## 最も重要な事実: データは「勝手に更新されない」

TanStack Query は **WebSocket やサーバープッシュではありません**。
サーバー側でデータが変更されても、クライアントは **能動的にフェッチしない限り古いデータを表示し続けます**。

```
サーバーでユーザーが追加された
  ↓
クライアントのキャッシュ: 古いまま
  ↓
「何かのトリガー」がない限り更新されない
```

### 更新のトリガー一覧

| トリガー | 条件 | 設定 |
|---------|------|------|
| コンポーネントのマウント | staleTime 超過時 | `refetchOnMount: true` |
| ウィンドウフォーカス | staleTime 超過時 | `refetchOnWindowFocus: true` |
| ネットワーク復帰 | staleTime 超過時 | `refetchOnReconnect: true` |
| ポーリング | 設定した間隔 | `refetchInterval: number` |
| 手動 refetch | 呼び出し時 | `refetch()` |
| invalidateQueries | 呼び出し時 | コード内で明示的に呼ぶ |

**つまり:** 別のユーザーがサーバーにデータを追加しても、上記のトリガーが発火するまでクライアントには反映されません。

---

## staleTime — 「新鮮」な時間

### staleTime とは

データ取得後、**この時間内は「新鮮(fresh)」とみなされ、再取得しない** 時間です。

```
staleTime: 30000 (30秒) の場合:

0秒:  fetchUsers() → データ取得 ✓   → キャッシュに保存
10秒: useQuery マウント            → キャッシュから返す（APIは叩かない）
20秒: タブに戻る                   → キャッシュから返す（まだ新鮮）
35秒: タブに戻る                   → キャッシュを返しつつ、バックグラウンドで再取得
```

### staleTime: 0 (デフォルト)

データは取得した瞬間に「古い(stale)」になります。
次にトリガーが発火したとき、常にAPIを叩きます。

```
0秒:  fetchUsers() → データ取得 ✓
1秒:  タブに戻る → バックグラウンドで再取得（データは即古い扱い）
```

### staleTime: Infinity

一度取得したら、明示的に `invalidateQueries` しない限り再取得しません。

### staleTime: 'static'

`Infinity` と同じですが、型が違います（v5で追加）。

### 推奨値の目安

| データの性質 | staleTime | 理由 |
|------------|-----------|------|
| 頻繁に変わるデータ（チャット等） | `0` | 常に最新が必要 |
| 時々変わるデータ（ユーザー一覧等） | `30_000` (30秒) | 多少の遅延はOK |
| ほぼ変わらないデータ（設定、マスタ等） | `300_000` (5分)〜 | めったに変わらない |
| 絶対に変わらないデータ | `Infinity` | APIスキーマ等 |

## gcTime — ガベージコレクション時間

### gcTime とは

コンポーネントがアンマウントされて **どのクエリからも使われなくなったキャッシュ** が、メモリに保持される時間です。

```
gcTime: 300000 (5分, デフォルト) の場合:

0秒:   /users ページ表示 → ユーザーデータ取得
60秒:  /dashboard に移動 → /users のコンポーネントがアンマウント
       → キャッシュはまだメモリに残っている（GCタイマー開始）
180秒: まだメモリに残っている
360秒: GCタイマー満了 → キャッシュがメモリから削除
420秒: /users に戻る → キャッシュなし → 再取得（ローディング表示）
```

### gcTime: 0 の場合

コンポーネントがアンマウントされた瞬間にキャッシュが消えます。
ページを戻るたびに必ずローディングが表示されます。

### staleTime vs gcTime の関係

```
staleTime: 「データが新鮮かどうか」の判定
gcTime:    「キャッシュをメモリに残す時間」

staleTime → 「APIを叩くかどうか」に影響
gcTime    → 「キャッシュからデータを即座に表示できるか」に影響
```

**重要:** `staleTime > gcTime` にしてはいけません。
キャッシュが削除された後も「新鮮」とみなされるのは矛盾します。

## refetch の種類と動作

### refetchOnMount

コンポーネントがマウントされたとき、データが stale なら再取得します。

```
true (デフォルト): stale なら再取得
'always':         staleTime に関係なく常に再取得
false:            マウント時に再取得しない
```

### refetchOnWindowFocus

ブラウザのタブに戻ったとき。

```
true (デフォルト): stale なら再取得
'always':         staleTime に関係なく常に再取得
false:            タブ復帰で再取得しない
```

### バックグラウンド再取得の仕組み

```
ユーザーがタブに戻る（refetchOnWindowFocus: true）

staleTime を超えていない場合:
  → 何もしない（キャッシュをそのまま表示）

staleTime を超えている場合:
  1. まずキャッシュのデータを即座に表示（ユーザーは古いデータを見る）
  2. バックグラウンドで API を叩く
  3. 新しいデータが返ってきたら、UI を更新
  → ユーザーには「データが一瞬で更新された」ように見える
```

## invalidateQueries の詳細動作

### 何が起きるか

```
queryClient.invalidateQueries({ queryKey: queryKeys.users.lists() })

1. マッチするすべてのクエリを「stale」とマークする
2. refetchType に応じて再取得を開始する
   - 'active'(デフォルト): 現在使用中のクエリだけ再取得
   - 'all': 使用中でないクエリも再取得
   - 'none': stale マークだけで再取得しない
```

### invalidateQueries vs refetchQueries

| メソッド | stale マーク | 再取得 |
|---------|------------|-------|
| `invalidateQueries` | する | active クエリのみ（デフォルト） |
| `refetchQueries` | しない | 指定したクエリを強制再取得 |

## キャッシュのライフサイクル図

```
1. queryKey が初めて使われる
   → Query エントリがキャッシュに作成される
   → status: 'pending', fetchStatus: 'fetching'

2. queryFn が成功
   → data がセットされる
   → status: 'success', fetchStatus: 'idle'
   → staleTime タイマー開始

3. staleTime 経過
   → isStale: true になる
   → 次のトリガー（マウント、フォーカス等）で再取得が可能に

4. コンポーネントがアンマウント（この Query を使う observer がゼロに）
   → gcTime タイマー開始
   → キャッシュのデータはまだメモリにある

5a. gcTime 経過前にコンポーネントが再マウント
    → キャッシュからデータを即返す
    → stale なら裏で再取得

5b. gcTime 経過
    → Query エントリがキャッシュから完全に削除
    → 次にマウントされたら最初からやり直し（ローディング表示）
```

## structuralSharing — 不要な再レンダリング防止

### これは何？

API レスポンスが **内容は同じだが参照が異なる** 場合に、前回のオブジェクト参照を再利用する機能です。

```ts
// 1回目のフェッチ: { users: [{ id: 1, name: 'Alice' }] }
// 2回目のフェッチ: { users: [{ id: 1, name: 'Alice' }] }  ← 内容は同じ

// structuralSharing: true (デフォルト) の場合:
//   → 2回目のデータは1回目と同じ参照を返す
//   → React は再レンダリングしない（Object.is で比較して同じ）

// structuralSharing: false の場合:
//   → 2回目のデータは新しいオブジェクト
//   → React は再レンダリングする（参照が異なるため）
```

**デフォルトで有効** なので、通常は気にする必要はありません。

## よくある誤解

### 誤解1: 「TanStack Query はリアルタイム更新ツール」

**事実:** TanStack Query は **キャッシュ管理ライブラリ** です。リアルタイム更新には WebSocket 等の別の仕組みが必要です。TanStack Query はフェッチ結果をキャッシュし、適切なタイミングで再フェッチするだけです。

### 誤解2: 「staleTime を長くすれば高速になる」

**事実:** staleTime を長くすると API呼び出しは減りますが、データが古くなるリスクが上がります。トレードオフです。

### 誤解3: 「invalidateQueries ですべてのデータが最新になる」

**事実:** `invalidateQueries` は表示中（active）のクエリだけ再取得します。表示されていないクエリは、次に表示されたとき初めて再取得されます。

### 誤解4: 「gcTime: 0 にすればメモリ節約」

**事実:** メモリ節約にはなりますが、ページ遷移のたびにローディング表示が出て UX が悪化します。適切な gcTime を設定しましょう。
